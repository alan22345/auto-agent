# Plan — ADR-016 Phase 4 (TypeScript + cross-language HTTP matching)

## Goal

Ship Phase 4 of ADR-016: a TypeScript tree-sitter parser, a Python decorator-handling fix, and a cross-language HTTP-matching stage that links `fetch`/axios calls to FastAPI/Flask route handlers. Pure addition — no behaviour change to Phases 1-3 except the documented Python decorator handling.

## Acceptance criteria

1. `tree-sitter-typescript>=0.23` is in `pyproject.toml`.
2. `agent/graph_analyzer/parsers/typescript.py` exists; `parsers/__init__.py` gains one registry entry (`.ts`, `.tsx`). `pipeline.py` requires NO change for TS parser dispatch.
3. `shared/types.py::Node` gains `decorators: list[str] = []`; `web-next/types/api.ts` regenerated to match.
4. Python parser descends into `decorated_definition` and populates `Node.decorators` for both functions and classes; line numbers reflect the wrapped definition.
5. `agent/graph_analyzer/http_match.py` exists: discovers FastAPI/Flask backend routes from Python `Node.decorators`, detects `fetch`/`axios` call sites in TS, and matches them by `(method, path_pattern)`. Unambiguous matches emit `Edge(kind="http", source_kind="ast")`; ambiguous matches go through one LLM call (validated via existing `validator.py`, edges tagged `source_kind="llm"`).
6. `pipeline.py` calls the HTTP matcher AFTER all per-area parsing + gap-fill; failure in HTTP matching is logged but never breaks the pipeline.
7. Existing Phase 1-3 unit tests still pass.
8. New tests cover: TS parser shapes (nodes, imports, inherits, calls, unresolved, syntax-error tolerance), Python decorators capture (function + class + line numbers), HTTP match (unambiguous AST, ambiguous LLM, path templating, no-match → no edge), end-to-end cross-lang pipeline.
9. `ruff check`, `ruff format --check`, web-next `tsc --noEmit`, `npm run build`, and `npx vitest run` all green.

## Tactical decisions / constraints

- **TS uses `.ts` and `.tsx`**; both go through the same parser (tree-sitter-typescript exposes `language_typescript()` and `language_tsx()`; we pick by extension internally).
- **Module ids for TS** follow Python's pattern: `module:<dotted>` with `/` → `.`. Relative imports stay as raw spec ids (`module:./foo`), absolute packages similarly (`module:react`).
- **Inherits** uses class id form `"{file}::{ClassName}"`, same as Python.
- **Statically resolvable calls**: bare identifier bound in module scope, `this.method()` whose receiver class defines `method`, `new Foo()` collapsed to `calls` of `Foo`, `Foo.staticMethod()` when `Foo` is a module-bound imported symbol.
- **HTTP method match is case-insensitive.** Path templating segments: anything inside `{...}` (FastAPI) or starting with `:` (Flask-ish) is treated as `*`. URL query strings stripped before match. Trailing slashes normalised.
- **LLM disambiguation prompt** mirrors `gap_fill.py` shape: one system prompt enumerating candidate route node ids + path patterns, one user message describing the frontend call, `complete_json` parses the JSON, emit one `Edge`. Validate citation against the **frontend** call line.
- **No `boundary_violation`** on HTTP edges in Phase 4 — Phase 5 owns that concept.

## Out of scope (Phase 5+)

- General cross-area edge flagging and `boundary_violation=True` (Phase 5).
- `query_repo_graph` tool, per-area refresh, UI distinction between AST and LLM edges (Phase 6/7).
- Template-string URLs in frontend calls (`` fetch(`/api/x/${id}`) ``) — left as unmatched in Phase 4; the parser still emits unresolved-site records for non-literal URLs.
- Reading `methods=[...]` from a Flask decorator in TypeScript-emitted decorators (there is no Flask in TS).
