# [ADR-016] Code graph — hierarchical, function-level, cited-edge map of an onboarded repo

## Status

Proposed

## Context

When a repo is added to auto-agent, the system does almost nothing structural with it. `POST /repos` creates a row in the `Repo` table and stops. `agent/harness.py` (when manually triggered via `/repos/{name}/onboard`) writes a CLAUDE.md, lint config, ADR templates and opens a PR — a *prose* analysis. `agent/context/repo_map.py` runs per-task to inject a flat text outline (file → classes → functions) into the system prompt. None of these produce a structural map that:

1. shows the repo's architecture visually,
2. is queryable by the agent at function-level granularity ("what calls X?", "what does this area depend on?"),
3. is grounded in the actual code with verifiable evidence,
4. can be inspected by a human alongside the agent.

The gap is most visible in two places. First, when the agent is asked to modify a symbol that has callers it can't see without grep-walking — it either does a wide grep (slow, noisy, blows context) or it misses callers (silent correctness bug). Second, when a human wants to understand an unfamiliar repo before pointing auto-agent at it — there's no quick way to grasp "what areas are there and how do they connect" without reading the codebase.

The asks that motivated this ADR were two distinct deliverables blurred together: "save findings to memory" (prose facts about the repo) and "create a visual graph of dependencies and how the project works." The first overlaps with existing harness output and is deferred. The second is what this ADR is about.

Three design tensions surfaced during the design grilling and shaped the decision:

**Tension 1 — Deterministic vs. LLM-derived.** Pure-AST graphs are deterministic but blind to dynamic dispatch (registry dicts, decorators, `getattr`, plugin loaders). Pure-LLM graphs cover those cases but aren't deterministic (sampling variance, context limits on large repos, hallucinated edges). Neither is correct alone for a function-level claim of "X calls Y."

**Tension 2 — One graph or two.** The agent wants exact structural queries (give me the callers of X). The human wants a navigable visualisation. The naive answer is two systems, two stores, two refresh paths. That guarantees drift.

**Tension 3 — Coupling to the existing repo lifecycle.** Folding graph analysis into `POST /repos` means every repo pays the cost (clone, parse, LLM gap-fill) whether the user cares or not. Decoupling means designing an opt-in surface from scratch.

## Decision

**Build a hierarchical, function-level, citation-validated code graph as an opt-in feature reachable from a new `/code-graph` route, backed by a dedicated table, generated by a structured pipeline with bounded agent-loop escape, and queryable by the agent through a single typed tool.**

Twelve load-bearing components. Each is locked by name.

### 1. Both agent and human are consumers; one source of truth

The graph data is the canonical store. The human reads it through a rendered visualisation; the agent reads it through a query tool. There is no separate "agent's version" of the graph and no separate "human's version." Two views over the same JSON.

This rules out duplicating the data into team-memory (which optimises for sparse high-signal facts, not dense mechanical edges). Team-memory remains untouched by this feature.

### 2. Hierarchical compound graph; function-level granularity; encapsulation by directory

Nodes nest. The top-level box is the repo. Below it are *areas*. Below each area are files. Below each file are classes (where applicable) and functions. Edges connect nodes at any level — and edges between fine-grained nodes are aggregated and shown at the level the user is currently looking at.

**Areas** default to top-level directories of the repo (`agent/`, `orchestrator/`, `shared/`, `integrations/`, `web-next/`, etc. for auto-agent itself). Repos that want a custom grouping declare it in `.auto-agent/graph.yml`:

```yaml
areas:
  - name: backend
    paths: ["src/api/**", "src/services/**"]
  - name: frontend
    paths: ["src/ui/**", "src/components/**"]
```

If the file is absent, top-level directories are used.

### 3. Hybrid analyser: tree-sitter skeleton, LLM gap-fill, citation validation

The pipeline has a deterministic spine. Tree-sitter produces the nodes (files, classes, functions) and the *cheap* edges that static analysis can resolve exactly: imports, inheritance, statically-resolvable calls. This is ~80–95% of the graph for typical code, fast, repeatable, free.

Dynamic-dispatch sites — registry dicts (`HANDLERS[event_type](payload)`), `getattr`-based routing, decorator-driven dispatch (FastAPI routes, Click commands), plugin systems — are detected by tree-sitter heuristics and then resolved by a **one-shot LLM call per site**. The LLM is given the relevant file slice and emits edges in a structured schema:

```json
{
  "source": "agent/loop.py::AgentLoop.run",
  "target": "agent/llm/bedrock.py::BedrockProvider.complete",
  "kind": "calls",
  "evidence": {"file": "agent/loop.py", "line": 203, "snippet": "response = await self.llm.complete(...)"}
}
```

**Every LLM-emitted edge is validated against the actual file.** The validator opens the cited file, reads the cited line, and checks the snippet matches. Edges whose citations don't validate are *dropped*, not silently kept. This is the load-bearing trust mechanism: the LLM is allowed to deduce, but it isn't allowed to fabricate without proof.

Each edge carries a `source_kind: "ast" | "llm"` tag so the UI can surface the distinction. The trust contract surfaced to the user is: **edges in the graph are a lower bound on reality, all verified, none invented.** Missing edges ≠ "no relationship" — they mean "static analysis + bounded LLM gap-fill couldn't prove this one."

### 4. Bounded agent-loop escape for low-confidence cases only

When a one-shot LLM call returns low-confidence output (no citations, citations all fail validation, refusal to enumerate), the pipeline can escalate that single site to a **bounded agent loop** with a narrow toolset (`read_file`, `grep`, `return_findings`), a max-iteration cap, and a max-time cap. The agent's findings are subject to the same citation validation as the one-shot path.

Agent-loop is never the default and never runs at the whole-graph level — only per-site, only when the cheaper path failed. This keeps the cost profile predictable.

### 5. Languages v1: Python and TypeScript; tree-sitter universal

The first ship targets Python and TypeScript because together they cover auto-agent itself end-to-end (Python in `agent/`/`orchestrator/`/`integrations/`; TypeScript in `web-next/`) and the majority of likely external repos. Tree-sitter is the universal AST layer for both — and for future languages added without rewriting the parser layer.

The cost named honestly: tree-sitter for TypeScript gives weaker static edges than `ts-morph` (which has full TS-compiler type info) would. We accept this for now because the LLM gap-fill is the same machinery in either language, and a future swap to `ts-morph` for the TS pipeline stage is a contained refactor.

### 6. Four edge kinds in v1: calls, imports, inherits, http

Every edge has a `kind: "calls" | "imports" | "inherits" | "http"`. Calls collapse instantiation (`new Foo()` → call edge to `Foo.__init__`/`Foo` constructor). Imports are module → module. Inherits is class → parent class. `http` is the cross-language edge — TypeScript `fetch`/axios call to a backend route handler, matched via URL pattern → route decorator. The matching itself is the highest-LLM-risk part of v1 and the most likely to need iteration.

### 7. Cross-area edges modelled explicitly; public surface; architecture violation flagging

Edges crossing area boundaries are first-class. Each area has a declared (or inferred) **public surface** — the set of symbols intended for cross-area consumption. Cross-area edges that target a public-surface symbol are valid; cross-area edges that reach into another area's internals are flagged `boundary_violation: true` in the schema, rendered with a warning indicator in the UI.

When a `.auto-agent/graph.yml` declares explicit boundary rules (e.g. `shared/ must not import from agent/`), violations of those rules get a stronger flag and surface in a "violations" panel on the graph page. For auto-agent itself, CLAUDE.md already spells out the dependency layering — the graph becomes a live drift detector for it.

### 8. Storage: dedicated `RepoGraph` table; team-memory not involved

Two new tables:

- **`RepoGraphConfig(repo_id PK, graph_enabled, analysis_branch, analyser_version, workspace_path, last_analysis_id, ...)`** — per-repo settings. Analysis branch defaults to the repo's `default_branch` and persists until changed.
- **`RepoGraph(id, repo_id, commit_sha, generated_at, analyser_version, status, graph_json, ...)`** — one row per completed analysis. `graph_json` carries the full nested-node + edge blob. Code snippets are *not* stored in the JSON; they're fetched lazily from the workspace on demand to keep the row size sane.

Team-memory remains the home for sparse high-signal facts (decisions, gotchas, conventions). The graph is dense mechanical data and is the wrong shape for the recall-by-relevance semantics team-memory provides.

### 9. Workspace: dedicated, persistent, branch-configured per repo

Each opted-in repo gets a persistent workspace at `graph-workspaces/<repo_id>/`. Separate from the harness workspace at `workspaces/<repo_id>/` because the harness workspace's HEAD moves as tasks check out feature branches, and mixing the two creates checkout contention.

Refresh = `git fetch origin && git reset --hard origin/<analysis_branch>` (safer than `pull` against force-pushes). A per-repo lock prevents concurrent refresh races. Branch is set on `RepoGraphConfig.analysis_branch`; changing it marks the existing graph as stale (banner displayed) but does *not* auto-refresh. The user decides when.

### 10. Pipeline: structured deterministic spine; failures isolated per area

New module `agent/graph_analyzer/` with submodules `parser.py`, `gap_fill.py`, `validator.py`, `cross_language.py`, `boundaries.py`, `pipeline.py`. Stages:

1. Acquire workspace lock → fetch + reset to configured branch → record `commit_sha`.
2. Per area: tree-sitter pass emits nodes + cheap edges.
3. Per area: detect dynamic-dispatch sites via tree-sitter heuristics.
4. Per site: one-shot LLM gap-fill (or bounded agent-loop escape) → cited edges.
5. Citation validation — drop unverifiable edges.
6. Cross-area edge identification + boundary-violation flagging.
7. Cross-language HTTP matching (tree-sitter finds fetch calls + route decorators; LLM matches URL patterns to routes).
8. Optional architecture-violation check against `.auto-agent/graph.yml`.
9. Assemble graph JSON → write `RepoGraph` row → publish `REPO_GRAPH_READY`.

Failures are isolated at the area level. If one area's analysis throws (parser error on a malformed file, LLM gap-fill exhausts retries), the rest still complete; the user sees a partial graph with a "refresh this area" button on the failed box.

### 11. UX: top-level `/code-graph` route; opt-in onboarding; expand/collapse rendering

A new top-level web-next route at `/code-graph`, listed in the sidebar alongside Tasks, Repos, Memory. Empty state prompts the user to onboard a repo (pick from existing `Repo` rows or add a new one — the "new repo" path reuses the existing `POST /repos` form so there's no duplicate UX).

Per-repo view at `/code-graph/{repo}`:

- **Library**: Cytoscape.js + `cytoscape-expand-collapse` extension.
- **Default view**: all areas collapsed. Shows top-level boxes + cross-area edges only.
- **Node click**: side panel with code preview (fetched lazily), incoming/outgoing edges grouped by kind.
- **Edge click**: evidence popover showing `(file, line, snippet)` and `ast`/`llm` source tag.
- **Controls**: search by name, edge-kind toggles, ancestor/descendant highlight.
- **Freshness banner**: "Graph from `<branch>`@`<sha>` analysed `<timestamp>` [Refresh whole graph] [Settings]". Per-box "refresh this area" button on each collapsed area.

### 12. Agent interface: one tool, multiple ops, staleness flags surfaced

New agent tool `query_repo_graph(repo_id, op, params)` with the following ops:

- `callers_of(node)` / `callees_of(node)`
- `outgoing_edges(node)` / `incoming_edges(node)`
- `public_surface(area)`
- `path_between(source, target)`
- `violates_boundaries(source, target)`

**Every response includes:**

- `staleness: {graph_sha, workspace_sha, drifted: bool}` — the agent's task workspace can be on a different branch from the analysis workspace; the tool reports the drift, the agent decides whether to trust the result.
- Per-result `exists_in_workspace: bool` — handles the case where the agent's task branch has deleted/renamed the symbol.

**Integration point:** `agent/context/system.py` adds a paragraph to the system prompt when the task's repo has `graph_enabled=true`, nudging the agent to consider `query_repo_graph` before grep-walking. No auto-injection of the graph itself into the prompt — that's deferred until we see which queries the agent reaches for most.

## Consequences

### What becomes easier

- **Agent gains a fast, exact, function-level structural query path.** Renames, refactors, and "what does this touch" investigations stop costing a wide grep.
- **Humans can land in an unfamiliar repo and grasp its shape in seconds.** Top-level collapsed view is the 10,000-foot map; expand into the area they care about.
- **Architecture drift becomes visible.** CLAUDE.md-stated boundaries are checkable against the live graph; violations surface.
- **Cross-language data flow becomes traceable.** Frontend-to-backend wiring is no longer purely in the developer's head.
- **Graph is the substrate for future features.** Impact analysis on PRs, automatic test selection by reachability, dead-code detection — all are graph queries.

### What becomes harder

- **A new piece of infrastructure to maintain.** Tree-sitter grammars, an LLM-gap-fill prompt, a citation validator, a graph schema, a UI rendering layer, an agent tool — each is a small surface, but together they're real ongoing cost.
- **The "graph is a lower bound" framing must stay visible.** If users start treating the graph as ground truth, missing edges will mislead. The UI's `ast`/`llm` source tagging and the "edges are statically + LLM-provable, not exhaustive" caveat in the freshness banner are load-bearing for user expectations.
- **Refresh latency.** First-time analysis of a real repo is minutes, not seconds. The freshness banner + per-area refresh mitigate but don't eliminate this.

### Specific risks named honestly

1. **Tree-sitter for TS is less precise than `ts-morph`.** TS edges from tree-sitter will have lower recall than they would with full TS-compiler type info. Mitigation: pipeline stage boundary means we can swap in `ts-morph` for the TS stage later without restructuring the rest.
2. **Cross-language HTTP matching via LLM is the v1 step most likely to land wrong.** Citation validation catches half of failures ("matched a route that doesn't exist"). The other half ("matched the wrong route because two share a prefix") will need iteration on the matching prompt + heuristic.
3. **Static call resolution in Python has known holes** that even the LLM gap-fill won't catch (highly-dynamic patterns, runtime monkey-patching). The "lower bound" framing absorbs this honestly; it must not silently become "graph claims completeness."
4. **Per-area refresh correctness depends on cross-area edge re-validation.** This is the highest-risk code path. If it has bugs, stale cross-area edges silently mislead. Strong test coverage on this specific path is required.
5. **Workspace contention.** Per-repo lock handles concurrent refresh; it does *not* handle the case where the analyser workspace is being scanned while the harness workspace for the same repo is being torn down or relocated. The two workspace trees are separate but they share the underlying repo URL/auth.

### Out of scope for v1 (deferred)

- **Prose "findings" extraction** as structured facts in team-memory. Existing harness output (CLAUDE.md, ADRs, lint config) is unchanged. If structured findings become useful, they're an additive feature on a different code path.
- **Auto-refresh on push / scheduled refresh.** Manual refresh only in v1. Avoids the "graph thinks it's fresh but isn't" failure mode.
- **Auto-injecting graph slices into task system prompts.** The system-prompt nudge points the agent at the tool; the tool's results land in context the standard way. Pre-fetching a relevant subgraph at task start is a future optimisation once we have signal on which queries dominate.
- **Branch picker that supports tags / arbitrary SHAs.** v1 = the configured branch HEAD only. Future flexibility is additive.

### Implementation phasing

1. `RepoGraphConfig` + migration + workspace scaffolding + per-repo lock.
2. Tree-sitter pipeline for one language (Python first); JSON schema + basic page that renders one repo's call+import graph.
3. LLM gap-fill + citation validation; agent-loop escape for low-confidence sites.
4. TypeScript pipeline; cross-language HTTP matching.
5. Cross-area edges + public surface + architecture-violation flagging.
6. `query_repo_graph` tool + system-prompt nudge in `agent/context/system.py`.
7. Per-area refresh + freshness banner + UI polish.

Each phase ships complete (no stubs, no "phase 1 fills it in"), is independently testable, and leaves the system in a deployable state.
